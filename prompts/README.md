# `prompts/` — the prompt artifact layer

This folder holds **ready-to-act Claude Code prompts** generated from `signals/` by the
`signal-to-prompt-bridge` loop. It is the bridge between *"a problem was found"* (a
signal) and *"here is the exact, correctly-stacked instruction to act on it"* (a prompt
a human chooses to run).

> **Generate, never execute.** The bridge loop only *writes* prompt files here. It never
> runs them, never edits source, never opens a source-changing PR. A human reads these
> and decides which to launch. This folder is a drafting output, not an execution queue.

## What IS a prompt artifact

- One self-contained Claude Code one-shot prompt, produced by the `oneshot-prompt-generator`
  skill, scoped to exactly one signal.
- Stack-resolved (banner from `STACK.md`) and evolution-mode tagged (band / gates).
- Classified by `action` (see below) so a reader instantly knows whether it's a ready fix
  or an investigation.

## What is NOT

- A fix, a diff, a PR — prompts describe work; running them is a separate human act.
- A signal — those live in `signals/` and are written by the audit loops, never here.
- A prompt for an unverified finding presented as a ready fix (see `action` rules).

## Naming

```
prompts/<YYYY-MM-DD>-<signal-id>-<short-slug>.md
```

## Schema

```markdown
---
id: prompt-<YYYY-MM-DD>-<nnn>
date: <YYYY-MM-DD>
source_signal: <signal id, e.g. sig-2026-06-25-006; aggregate item: sig-...-010#item-slug>
signal_status: confirmed | needs-human
band: 0 | 1 | 2 | 3
gates: required | recommended | optional
action: execute | investigate
---

> Target stack: <resolved from STACK.md>

[the full generated one-shot prompt body]
```

## `action` — the load-bearing field

| `signal_status` | `action` | meaning |
|---|---|---|
| `confirmed` | `execute` | the verifier confirmed the finding. The prompt is a **ready fix** — run it when you choose. |
| `needs-human` | `investigate` | the verifier **could not confirm**. The prompt leads with the open question, instructs *verify-the-finding-is-real first*, and its "Done When" is a **decision, not a code change**. NEVER a ready fix. |

**The mapping is strict and never crossed.** A `needs-human` signal can only ever produce
an `action: investigate` prompt. Confusing the two is the single way this loop could cause
harm (someone runs an unverified finding as a fix), so it is enforced at generation time.

## Pitchey gate convention

Because the worker is **never type-checked in CI** (esbuild bundles without checking), an
`execute` prompt for a worker type fix carries a **`Done When` green gate** of
`./frontend/node_modules/.bin/tsc --noEmit -p tsconfig.json` showing the cited error gone
**and no new errors introduced** — re-running the very check that found it. High-severity
live-path fixes are tagged `gates: recommended`; low-severity cosmetic ones `optional`.

## Dedup & repeat runs

Dedup by `source_signal`. If a prompt already exists for a signal (or aggregate item),
do not regenerate it — append a dated note to its body only if the underlying signal
changed. New signals → new prompts.

## Provenance

Prompts are generated by [`signals-to-prompts`](../.claude/skills/signals-to-prompts/SKILL.md)
for the [`signal-to-prompt-bridge`](../loops/signal-to-prompt-bridge/README.md) loop, which
runs after the audit loop(s) so it reads the freshest signals.
