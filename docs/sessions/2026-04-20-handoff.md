### #26 — release-management.yml — recommended decision: delete

Evidence for the delete path:
- **10 of the last 10 runs have failed.** Nobody noticed because `simple-health-check.yml`'s alerting (Slack + GitHub issue-creation) only fires on **runtime** health failures, not on **CI workflow** failures. The alerting-with-teeth from PR #3's observability hardening doesn't cover CI status.
- **Two dead file references** in the workflow itself:
  - Line 306–307 references `src/worker-production-db.ts` which doesn't exist (grep confirms). That `sed` step is a no-op silently.
  - Line 564 references `.github/workflows/cloudflare-deploy.yml` which doesn't exist either. The `trigger-deployment` step can never have fired successfully.
- **A separate shell-escaping bug** in the changelog construction (lines 443–475): `RELEASE_NOTES="..."` is double-quoted and the changelog content contains backticks (from ```` ```bash ```` deployment-command blocks), which bash interprets as command substitution. The actual failure message is `File name too long` because the backtick-substituted content overflows a filename-length limit when bash tries to resolve it.
- **Actual deploys** happen via `🚀 Pitchey Platform CI/CD` (`.github/workflows/ci-cd.yml`) and `Deploy Frontend to Cloudflare Pages` (`.github/workflows/deploy-frontend.yml`). Release-management is redundant with both and doesn't block anything.
- **GitHub releases** are the only artifact it uniquely produces — and the last successful one was never. There's no release history being lost.

Alerting-gap follow-up needed after the delete: either extend `simple-health-check.yml` to additionally assert "no workflow failed in the last 24h" (low effort, high signal), or accept that CI-failure visibility lives only in GitHub Actions UI and document that choice. This is a #20-adjacent problem: failure modes that don't appear anywhere monitored are how drift starts.
