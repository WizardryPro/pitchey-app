_Generated by the `roadmap-from-live-code` ultracode workflow: 12 subsystem explorers grounded in live code (file:line evidence) → strategic synthesis → run-ready prompts. 29 agents._

## Headline

Pitchey is feature-complete for a demand-first launch; the next phase is not more features but de-risking the two things that actually break it — Neon compute-quota outages and silent side-effect failures behind green tests — then deepening the one defensible asset (the cross-role NDA-intent + thesis graph) into something observable and matchable. Sequence: kill the known runtime-broken paths (credit constraint, unverified auth/payment schema) before launch, harden reliability and side-effect testing, then invest moat capital into thesis-matching visibility and NDA-graph liquidity metrics.

### ~~R1~~ — Fix credit_transactions type='bonus' CHECK-constraint violation — ✅ RESOLVED (false alarm)
**Status:** Closed 2026-06-30 — **not a bug.** Verified against live prod (Neon `patient-surf-83998605`, default branch). No code change shipped (per the prompt's decision rule: don't ship a no-op migration claiming a fix).

**What G1 found (the live truth):**
- `credit_transactions.type` is a Postgres **ENUM** `credit_transaction_type`, **not** a `VARCHAR+CHECK`. There is **no CHECK constraint** on the table (`pg_constraint` returned empty).
- Enum values: `purchase, usage, refund, bonus`. **`'bonus'` is permitted; `'grant'` does not exist.**
- **28 `'bonus'` rows already exist in prod** — the inserts at `worker-integrated.ts:2033/21835/22768` succeed today.
- No live code inserts `'grant'` (only migration 038's text mentions it), so the inverse failure mode is also absent.

**Why the roadmap mis-flagged it:** the subsystem explorers read migration `038_credit_system.sql:20` (`CHECK (type IN (...,'grant'))`) and marked R1 "grounded: true / fails at runtime today." But prod was hand-migrated (baselined 2026-04-17) and diverged from the file. The memory note [[project_credit_type_drift]] was correct; the migration file is the drift. This is the #20 pattern — and the verify gate caught it before any change.

**Real residue (NOT a launch blocker):** migration-file drift only. `038` would build a `VARCHAR+CHECK(...,'grant')` schema that contradicts prod's enum — this only bites a from-scratch rebuild from migrations (disaster recovery / fresh env). Track under R20/disaster-recovery hygiene, not Band 0. Do not edit `038` in place (already baselined; changing its SHA trips the deploy gate).

## Band 1 — Operational reliability & silent-breakage

_The two biggest non-feature risks per every map and project memory: Neon compute-quota 402 has taken prod down twice with no proactive detection, and the test suite asserts HTTP status not side-effects so handlers can return 200 while dropping writes. This band turns invisible failure classes into observable, alertable, test-caught ones before traffic scales at launch._


### R6 — Proactive Neon compute-quota detection + alerting + keep-warm decision
**Risk:** 🟡 med  ·  **Effort:** M

**Why:** Quota exhaustion 402s every query → login 503s, marketplace silently empty (errors swallowed). Happened 2026-04-30 (#65) and 2026-06-22. Observability map: no quota probe exists; KEEP_WARM_DB is gated OFF. simple-health-check is hourly (up-to-1h latency). This is the single highest operational risk for a launch.

**Scope:** Add quota/compute headroom probe to /api/health (degraded before exhaustion), wire to Slack + GitHub issue alert path (already exists for health-check), and make a documented plan decision on KEEP_WARM_DB + Neon plan tier. Files: worker-integrated.ts health handler, simple-health-check.yml, wrangler.toml KEEP_WARM_DB.

**Moat impact:** Neutral — keeps the platform up; the moat is worthless during an outage.

**Verification gates:**
- G1: Distinguish Neon 402 (compute quota) from HTTP 402 INSUFFICIENT_FUNDS (credits) in the probe — observability map notes the codebase conflates these
- G2: Verify alert actually pages (test the Slack/issue path with a synthetic degraded response)
- G3: Confirm keep-warm cron's own DB pings don't accelerate quota burn (the reason it was gated off) before enabling

### R7 — Side-effect verification in the integration test tier
**Risk:** 🟢 low  ·  **Effort:** L

**Why:** Confirmed across NDA, trust, collaboration, payments, data-quality maps: integration tests assert status + cookie presence, never the DB write-back. A handler can 200 while the NDA isn't created, credits aren't charged, or verification_tier isn't mutated. This is THE silent-breakage root cause and the cheapest insurance against the bug class that hid the consumption-gate failure for weeks.

**Scope:** Extend test/integration/*.test.ts with round-trip assertions: POST NDA request → assert ndas row + credit decremented; publish pitch → assert seal row + matching-investor notification rows; identity verify → assert verification_tier mutated. Files: test/integration/, vitest.backend.config.ts.

**Moat impact:** Core — the NDA/seal/verification side-effects ARE the moat data; untested they can silently stop accruing.

**Verification gates:**
- G1: Tests run against a throwaway Neon branch with the no-prod guard (TEST_DATABASE_URL) — never prod
- G2: Each new test must fail if the side-effect is removed (mutation-test the assertion by temporarily disabling the write)
- G3: Prioritize money/NDA/verification paths first; do not chase coverage % on read endpoints
